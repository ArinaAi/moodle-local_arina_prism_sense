# Arina.ai Plugin Registration Flow

**Version:** 2.0  
**Date:** 14 April 2026  
**Plugin:** `local_arina_prism_sense`  
**Scope:** Standalone Moodle installs registering with `arina.ai` to obtain an `api_key` and `org_id`.

---

## Overview

The registration flow connects a freshly installed Moodle plugin to the `arina.ai` backend. It uses:

- **Email verification** to prove identity
- **Pure polling** — Moodle asks `arina.ai` *"is this session done yet?"* every 5 seconds. No callbacks, no redirects, no exchange tokens.
- **No new DB table** — state is stored in Moodle's existing `mdl_config_plugins` table via `set_config`
- **Works on localhost** — because Moodle is always the one making requests outward to `arina.ai`, never the other way around

### What Changed From Previous Design

| Old (v1) | New (v2) |
|----------|----------|
| `arina.ai` redirected browser back to `register.php` | ❌ Removed — no redirect at all |
| `exchange_token` passed in URL | ❌ Removed — credentials never touch a URL |
| `register.php` redeemed token server-to-server | ❌ Removed — `register.php` no longer needed |
| `callback_url` sent to `arina.ai` | ❌ Removed |
| Polling checked local `set_config` state | ✅ Polling now calls `arina.ai` directly (via PHP proxy) |
| `arina.ai/verify-email` redirected to Moodle | ✅ Now shows a static success page on `arina.ai` |

### State Keys Written to `mdl_config_plugins` (plugin = `local_arina_prism_sense`)

| Key | Values | Purpose |
|-----|--------|---------|
| `reg_status` | `pending` / `complete` / `failed` | Current registration state |
| `reg_session_token` | UUID v4 string | **Generated by `arina.ai`.** Returned in the `/initiate-registration` response. Used by polling to identify this specific registration session. |
| `reg_timecreated` | Unix timestamp | Used to detect client-side timeout (15 min) |
| `reg_error` | Error string or empty | Shown to admin on failure |
| `api_key` | `sk-xxx` | Saved on success |
| `org_id` | `org-yyy` | Saved on success |
| `org_wallet_owner_id` | `org-yyy` | Saved on success — same value as `org_id`. Used by the credit service wallet lookup. Stored as a separate key to match the IOMAD `local_arina_prism_sense_company_config.org_wallet_owner_id` column naming. |

## Scenario 1 — The Register Button

### Trigger
Admin is on **Site Administration → Plugins → Local Plugins → PRISM Sense** (`settings.php`).  
The `api_key` config key is empty. A **"Register with Arina.ai"** button is visible.

### Steps

```
Admin clicks "Register with Arina.ai"
        │
        ▼
JS fires AJAX POST → /local/arina_prism_sense/api/initiate_registration.php
        │
        ▼
PHP (initiate_registration.php):
  1. Reads $USER->email, $USER->username, fullname($USER) from Moodle session
  2. Stores time() in $SESSION->arina_reg_last_sent  (for resend cooldown)
  3. Calls arina.ai/initiate-registration  (see API contract below)
  4. Receives session_token from arina.ai response
  5. set_config('reg_status',        'pending',      'local_arina_prism_sense')
     set_config('reg_session_token', session_token,  'local_arina_prism_sense')
     set_config('reg_timecreated',   time(),         'local_arina_prism_sense')
     set_config('reg_error',         '',             'local_arina_prism_sense')
  6. Stores session_token in $SESSION->arina_reg_session_token
  7. Returns JSON to JS
```

### API Call — `POST arina.ai/initiate-registration`

**Request (PHP → arina.ai):**
```json
{
  "email":     "admin@their-moodle.com",
  "username":  "admin",
  "full_name": "John Doe"
}
```

> No `session_token` in the request — `arina.ai` generates it and returns it.

**Response (arina.ai → PHP) — Success `HTTP 200`:**
```json
{
  "status":        "email_sent",
  "session_token": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

**Response — Failure `HTTP 4xx/5xx`:**
```json
{
  "status":  "error",
  "message": "Human-readable error description"
}
```

### AJAX Response (PHP → JS)

**Success:**
```json
{
  "status":        "email_sent",
  "email":         "admin@their-moodle.com",
  "session_token": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

**Failure:**
```json
{
  "status":  "error",
  "message": "Could not reach Arina.ai. Please check your internet connection."
}
```

### JS Behaviour After Response

| Response | JS Action |
|----------|-----------|
| `email_sent` | In-place swap: hide Register button, show waiting screen (Scenario 2). Start polling loop (Scenario 3). |
| `error` | Show inline error message below the Register button. Button stays active. |

## Scenario 2 — The Resend Email Button

### Context
The admin is on the **waiting screen** (Register button has been replaced in-place).  
The waiting screen shows:

- ✉️ *"A verification email has been sent to **admin@their-moodle.com**. Open your email and click the verification link to complete setup."*
- A **"Resend Email"** button — **disabled** on first render, counting down: *"Resend in 60s … 59s …"*

### Resend Button State Machine

```
Page loads with waiting screen → button DISABLED → countdown 60s
                                                         │
                                                  countdown hits 0
                                                         │
                                                         ▼
                                                 button ENABLED ("Resend Email")
                                                         │
                                               Admin clicks "Resend Email"
                                                         │
                                                         ▼
                                 JS fires AJAX POST → api/initiate_registration.php
                                   { "action": "resend" }
                                                         │
                                          PHP checks $SESSION->arina_reg_last_sent
                                                         │
                           ┌─────────────────────────────┴───────────────────────────┐
                           │ < 60s elapsed                                            │ ≥ 60s elapsed
                           ▼                                                          ▼
                 return cooldown response                          Re-call arina.ai/initiate-registration
                 { status: "cooldown",                             with SAME session_token, new email sent
                   remaining: N }                                  Update $SESSION->arina_reg_last_sent = time()
                           │                                                          │
                 JS syncs countdown                                       return { status: "email_sent" }
                 to remaining value                                                   │
                                                                          JS resets countdown to 60s
                                                                          button DISABLED again
```

### API Call on Resend — `POST arina.ai/initiate-registration`

**Identical request shape to Scenario 1** — no `session_token` needed:
```json
{
  "email":     "admin@their-moodle.com",
  "username":  "admin",
  "full_name": "John Doe"
}
```

`arina.ai` behaviour on receiving a duplicate email:
- Looks up the existing session by `email`
- Invalidates the previous `verify_token` for this session (if still active)
- Generates a fresh `verify_token` with a new 10-minute TTL
- Sends a new verification email
- Returns the **same `session_token`** as before (session is reused, not replaced)

> **The `session_token` stored in Moodle `set_config` does not change on resend.** The polling loop continues uninterrupted.

### AJAX Response to JS

**Cooldown (< 60s since last send):**
```json
{
  "status":    "cooldown",
  "remaining": 42
}
```

**Resent successfully (≥ 60s):**
```json
{
  "status": "email_sent",
  "email":  "admin@their-moodle.com"
}
```

## Scenario 3 — The Polling Check

### Overview

While the waiting screen is visible, JS polls every **5 seconds**:

```
GET /local/arina_prism_sense/api/check_registration_status.php
    ?session_token=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

PHP acts as a **proxy** — it forwards the `session_token` to `arina.ai/check-registration-status` and returns the result to JS. The `arina.ai` URL is never exposed to the browser.

PHP also enforces the **15-minute client-side timeout** by checking `reg_timecreated` before forwarding.

### Polling Flow

```
JS polls every 5s → api/check_registration_status.php?session_token=xxx
                              │
                   PHP reads reg_timecreated from set_config
                              │
              ┌───────────────┴───────────────────────────┐
              │ time() - reg_timecreated >= 900 (15 min)  │ still within 15 min
              ▼                                            ▼
       return { status: "expired" }           POST arina.ai/check-registration-status
                                                { session_token: "xxx" }
                                                          │
                                          ┌───────────────┼───────────────────┐
                                          │               │                   │
                                     "pending"        "complete"           "failed"
                                          │               │                   │
                                   return as-is    PHP writes:          PHP writes:
                                                   set_config            set_config
                                                   api_key               reg_status=failed
                                                   org_id                reg_error=message
                                                   reg_status=complete
                                                   return { status: "complete" }
```

### API Call — `POST arina.ai/check-registration-status`

**Request (PHP → arina.ai):**
```json
{
  "session_token": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

**Response — Still waiting `HTTP 200`:**
```json
{
  "status": "pending"
}
```

**Response — Verified and ready `HTTP 200`:**
```json
{
  "status":  "complete",
  "api_key": "sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  "org_id":  "org-yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy"
}
```

**Response — Verification failed `HTTP 200`:**
```json
{
  "status":  "failed",
  "message": "Human-readable reason"
}
```

**Response — Unknown session `HTTP 404`:**
```json
{
  "status":  "failed",
  "message": "Session not found or expired"
}
```

> `arina.ai` returns `api_key` and `org_id` directly in the `complete` response. They travel over HTTPS server-to-server (PHP → arina.ai) and are **never exposed in the browser or any URL**.

### Poll Response Shape Returned to JS

```json
{
  "status":  "pending" | "complete" | "failed" | "expired",
  "message": "Optional — only on failed or expired"
}
```

---

### 3a — Status: `pending` (Waiting)

**Condition:** `arina.ai` returns `{ "status": "pending" }` AND within 15-min window

**PHP Response to JS:**
```json
{ "status": "pending" }
```

**JS Behaviour:** Do nothing. Keep polling every 5 seconds.

---

### 3b — Status: `complete` (Success)

**Condition:** `arina.ai` returns `{ "status": "complete", "api_key": "...", "org_id": "..." }`

PHP writes to `set_config`:
```
api_key              = sk-xxx
org_id               = org-yyy
org_wallet_owner_id  = org-yyy  ← same value as org_id
reg_status           = complete
```

**PHP Response to JS:**
```json
{ "status": "complete" }
```

**JS Behaviour:**
1. Stop polling loop immediately
2. Redirect to: `settings.php?section=local_arina_prism_sense&registered=1`
3. Settings page detects `?registered=1` → shows success notification:  
   *"✅ Registration successful! Your API Key and Org ID have been saved."*
4. Register button area replaced with:  
   *"✅ Registered — Org ID: `org-yyy` · API Key: `••••xxxx`"*

---

### 3c — Status: `expired` (15-Minute Timeout)

**Condition:** `time() - reg_timecreated >= 900` — checked by PHP before calling `arina.ai`

**PHP Response to JS:**
```json
{
  "status":  "expired",
  "message": "Registration session timed out. Please try again."
}
```

**JS Behaviour:**
1. Stop polling loop
2. Show inline error: *"Your registration session timed out. Please click Register to try again."*
3. Restore the **Register button** — clear the waiting screen
4. PHP cleans up stale config keys on the next Register click

---

### 3d — Status: `failed` (Hard Failure)

**Condition:** `arina.ai` returns `{ "status": "failed", "message": "..." }` or `HTTP 404`

PHP writes:
```
reg_status = failed
reg_error  = <message from arina.ai>
```

**PHP Response to JS:**
```json
{
  "status":  "failed",
  "message": "Email verification failed. Please click Resend Email and try again."
}
```

**JS Behaviour:**
1. Stop polling loop
2. Show inline error: *"[message]"*
3. Re-enable **Resend Email** button immediately — bypass the 60s countdown (admin needs to act)
4. Register button also becomes available

## Scenario 4 — Admin Clicks the Email Link

### What is `verify_token`?

When `arina.ai` stores the registration intent (after `/initiate-registration`), it internally generates a `verify_token` and embeds it in the verification email link. This token is:

- **Generated by `arina.ai`** — Moodle never sees it
- **Stored by `arina.ai`** in its own DB, mapped to the `session_token`
- **Used only by `arina.ai`** to confirm the person clicking the link is the one who owns the email
- **Single-use, 10-minute TTL** — purely `arina.ai`'s internal concern

It works exactly like a password reset link — the token in the URL proves email ownership. Moodle has zero involvement with it.

### Trigger
Admin opens the verification email and clicks the link:  
`https://arina.ai/verify-email?token=<verify_token>`  
*(link is built and sent entirely by `arina.ai` — Moodle never constructs or sees this URL)*

### arina.ai Backend Steps

```
Browser hits arina.ai/verify-email?token=<verify_token>
        │
        ▼
arina.ai validates verify_token
(exists, not expired, not already used)
        │
        ▼
Creates user account using stored { email, username, full_name }
Generates org_id and api_key
Stores { api_key, org_id } against session_token in its DB
Marks verify_token as used (single-use)
        │
        ▼
Renders arina.ai-hosted success page:
┌──────────────────────────────────────────┐
│  ✅  Email Verified!                     │
│                                          │
│  Your Arina.ai account is ready.         │
│                                          │
│  Return to your Moodle settings page —  │
│  it will update automatically.           │
└──────────────────────────────────────────┘
```

- **No redirect to Moodle.** No `callback_url`. No exchange token.
- The admin sees a clean `arina.ai`-hosted page and can close the tab.
- The `settings.php` polling loop detects `complete` on its next tick (within 5 seconds) and updates automatically.

### What Happens If `verify_token` Is Expired (> 10 min)

`arina.ai` renders an error page:
```
┌──────────────────────────────────────────┐
│  ❌  Link Expired                        │
│                                          │
│  This verification link has expired.     │
│  Return to your Moodle settings page    │
│  and click "Resend Email" to get a new  │
│  link.                                   │
└──────────────────────────────────────────┘
```

The `session_token` stays `pending` on the Moodle side. It will timeout after 15 minutes and the polling loop will return `expired`, prompting the admin to restart.

## Scenario 5 — Page Reload / Return to Settings During Pending

### Trigger
Admin refreshes `settings.php` or navigates away and comes back while `reg_status = pending`.

### PHP Behaviour on Settings Page Load

On `settings.php` load, PHP checks:

```php
$regStatus      = get_config('local_arina_prism_sense', 'reg_status');
$regCreated     = get_config('local_arina_prism_sense', 'reg_timecreated');
$sessionToken   = get_config('local_arina_prism_sense', 'reg_session_token');
$isStillPending = ($regStatus === 'pending') && ((time() - $regCreated) < 900);
```

| Condition | Settings Page Shows |
|-----------|---------------------|
| `reg_status = complete` (api_key set) | ✅ Registered state — Org ID + masked API Key |
| `reg_status = pending` AND < 15 min old | Waiting screen auto-restored. Polling resumes with existing `session_token`. Resend countdown starts fresh (60s). |
| `reg_status = pending` AND ≥ 15 min old | Register button shown. Stale state silently cleaned up. |
| `reg_status = failed` | Register button shown. Error message displayed. |
| `reg_status` empty / not set | Register button shown (initial state) |

## arina.ai Backend — Endpoint Summary

| Endpoint | Method | Called By | Purpose |
|----------|--------|-----------|---------|
| `/initiate-registration` | `POST` | `api/initiate_registration.php` (PHP server-to-server) | Stores registration intent, sends verification email |
| `/verify-email` | `GET` | Admin's browser (via email link) | Verifies email, creates account + `org_id` + `api_key`, stores against `session_token`, shows static success page |
| `/check-registration-status` | `POST` | `api/check_registration_status.php` (PHP server-to-server) | Returns `pending` / `complete` (with credentials) / `failed` for a given `session_token` |

> **`/redeem-token` and `callback_url` are not needed.** Removed entirely.

### Validation Rules on `arina.ai`

| Rule | Detail |
|------|--------|
| `session_token` must be a valid UUID v4 | Basic format validation |
| `verify_token` TTL | 10 minutes — single-use |
| `session_token` result TTL | `arina.ai` should keep the `complete` result (with `api_key` + `org_id`) available for polling for at least **20 minutes** to cover the 15-min Moodle window |
| `/check-registration-status` rate limit | Accepts up to ~20 req/min per `session_token` (covers 5s polling comfortably) |

## Moodle Plugin — File Summary

| File | Role |
|------|------|
| `settings.php` | Register button UI, AJAX calls, in-place swap, polling loop, resend countdown, page-load state restore |
| `api/initiate_registration.php` | AJAX endpoint — generates `session_token`, writes `set_config`, calls `arina.ai/initiate-registration`, manages resend cooldown |
| `api/check_registration_status.php` | Polling proxy — checks 15-min timeout, calls `arina.ai/check-registration-status`, writes `api_key` + `org_id` + `reg_status` to `set_config` on `complete`, returns status to JS |
| `lang/en/local_arina_prism_sense.php` | New string keys for all registration UI text |
| `version.php` | Bumped to `2026041400` |

> **`register.php` is no longer needed for the registration flow.** It can be kept as a legacy stub or removed.  
> **No new DB table required.** All registration state lives in `mdl_config_plugins` via `set_config` / `get_config`.
